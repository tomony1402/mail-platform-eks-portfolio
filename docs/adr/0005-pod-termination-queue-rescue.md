# ADR 0005: Pod 終了時のキュー消失を3層防御で処理する

## ステータス

採択済み — 2026-04-27（実測データに基づき Phase アプローチから方針変更）

## コンテキスト

Postfix Pod は tmpfs（メモリ上）にキューを保持しているため、Pod 終了時にキューが消失する可能性がある。

消失の発生源は主に3つ：

1. **expireAfter（40分/50分）**
   Karpenter の NodeClaim expireAfter 設定により、
   deployment-a は40分、deployment-b は50分でノードが自発的に置き換わる

2. **Karpenter consolidation/drift**
   Karpenter が空きリソース統合や設定変更時にノードを置き換える

3. **AWS Spot 中断（2分前通知）**
   AWS が Spot インスタンスを回収する際の2分間の猶予のみ

## 計測結果

admin-server の計測基盤で実データを確認した結果：

- **終了時キュー件数**：平均5件・最大11件
- **Spot 中断**：過去5日で0回

キューは全然詰まっていない。Postfix Pod は健全に配信しており、キューが溜まる前に終了している。
この規模であれば preStop で数秒待つだけで大半が配信完了する。

## 決定

**3層防御アーキテクチャを採択する。**

3つの層はそれぞれ独立して動作し、連携はしない。
動作タイミングも救える規模も異なる。第2・第3層は既存の仕組みを流用するため、追加インフラは不要。

### 第1層：preStop でキュー配信完了待ち（新規実装）

**動作タイミング：Pod 終了時のみ**

Pod 終了前に最大25秒間、キューが空になるまで待機する。
数秒〜数十件レベルのキューを消失ゼロで終了させることを目的とする。

- 平均5件・最大11件の実績から、25秒あれば大半のケースでキューが空になる
- タイムアウト超過・コマンド失敗でも Pod 終了を止めない（常に `exit 0` で抜ける）

### 第2層：queue-monitor で S3 退避（既存）

**動作タイミング：常時（定期実行）、Pod 終了とは無関係**

キューが1,000件を超えた場合に S3 へ退避する。
通常運用中の大量滞留（障害・配信詰まり）を救うことを目的とする。
queue-monitor が20並列でアップロードし、Pod が終了してもデータは残る。

### 第3層：s3-recovery で再配信（既存）

**動作タイミング：常時（定期実行）、Pod 終了とは無関係**

S3 に退避されたメールを別 Pod から再配信する。
第2層とセットで機能するが、第1層とは独立している。
CronJob で定期的に実行される。

## 構成図

```
【Pod 終了時】
  SIGTERM
    ↓
  [第1層] preStop: postqueue が空になるまで最大25秒待機
    ・キューが空 → そのまま終了（消失ゼロ）
    ・タイムアウト/失敗 → そのまま終了（残キューあり、消失の可能性）

【常時（Pod 終了とは独立して定期実行）】
  [第2層] queue-monitor: キュー1,000件超で S3 に退避（CronJob）
  [第3層] s3-recovery: S3 退避済みメールを再配信（CronJob）

※ 第1層と第2・第3層は連携しない。救える規模・タイミングが異なる別の防御層。
```

## この方針が妥当な理由

| 観点 | 根拠 |
|:---|:---|
| データに基づく設計 | 推測ではなく実測値（平均5件・最大11件）で判断 |
| 追加インフラなし | 第2・第3層は既存の仕組みを流用 |
| シンプルな実装 | 第1層は数行の shell script で完結 |
| フェールセーフ | 第1層の失敗が Pod 終了を阻止しない |

## 実装

### preStop スクリプト

```yaml
lifecycle:
  preStop:
    exec:
      command:
      - /bin/sh
      - -c
      - |
        DEADLINE=$(($(date +%s) + 25))
        while [ "$(date +%s)" -lt "$DEADLINE" ]; do
          postqueue -p 2>/dev/null | grep -q "Mail queue is empty" && exit 0
          sleep 3
        done
        exit 0
```

動作：
- `postqueue -p` でキュー状態を確認し、"Mail queue is empty" が出たら即座に終了
- 3秒ごとに再チェック、最大25秒（= 約8回チェック）
- タイムアウトしても `exit 0` で抜けるため Pod 終了は阻害しない

### 対象ファイル

- `manifests/postfix/deployment-a.yaml`
- `manifests/postfix/deployment-b.yaml`

## 時間制約の整合性

```
terminationGracePeriodSeconds（現状30秒）= preStop 最大25秒 + Postfix プロセス終了処理
```

Spot 中断（t=0）から AWS 強制停止（t=120秒）まで、現状の30秒 grace period で第1層は十分動作する。

| grace 設定 | preStop に使える時間 | 判定 |
|:---:|:---:|:---:|
| 30秒（現状） | 最大25秒 | OK（5秒の余裕） |
| 90秒 | 最大25秒 | OK（将来の拡張余地） |

## 残るリスク

3層防御を採択した上で、いずれの層もカバーできないケースが存在する。

### カバーできないケース：終了時キューが数百件

```
Pod 終了時にキューが数百件ある場合：
  第1層 preStop 25秒 → 捌ける件数に限界（〜数十件）
  第2層 queue-monitor → 閾値1,000件未満のため動作しない
  → 残キューは消失
```

### 発生確率

実データ（平均5件・最大11件）から、現時点での発生確率は**極めて低い**。
ただしゼロではなく、以下の状況で発生しうる：

- 外部宛先の一時障害でキューが積み上がっている最中に Pod が終了した場合
- 大量送信バースト中に Pod が終了した場合

### 将来の対策候補

| 対策 | 効果 |
|:---|:---|
| preStop で残キューを S3 退避するロジック追加 | 数百件の消失をゼロにできる |
| queue-monitor の閾値を下げる（例：100件） | 大量滞留の検知を早める |
| preStop の timeout を延長（25秒→90秒） | 捌ける件数を増やす |

現時点では実データが低件数のため対応しない。キューが常態的に増加し始めた時点で対策を検討する。

## 将来の拡張条件

以下の条件が現れた場合のみ追加対策を検討する：

| 条件 | 対応 |
|:---|:---|
| 終了時キューが50件超えを常態化 | `terminationGracePeriodSeconds` を90秒に延長 |
| Spot 中断が月10回超え | Karpenter SQS interruption queue を導入 → **2026-05-13 導入済み** |
| s3-recovery の遅延が問題化 | SQS + dead-letter queue に移行 |

## 運用課題（検討過程で発見）

### Karpenter SQS interruption queue
2026-05-13 導入済み。Spot中断0回の段階で予防的に設定。月額コスト：約 ¥100。
導入の詳細・構成・理由は [ADR 0006](./0006-karpenter-sqs-interruption-queue.md) 参照。

### HELO ConfigMap 二重管理
`postfix-helo-script`（default namespace）と `recovery-helo-script`（kube-system namespace）が別管理されており、
ダッシュボードからの更新時に後者が同期されていなかった。
2026-04-24 に admin-server.py に同期処理を追加して暫定対応済み。

### s3-recovery CronJob の Git 管理
本番クラスタに存在するが、リポジトリで管理されていなかった。
2026-04-24 にマニフェストをリポジトリに追加済み。
