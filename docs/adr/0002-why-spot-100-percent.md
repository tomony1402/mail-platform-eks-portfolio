# ADR 0002: なぜ Spot 率100% にしたか

## ステータス
承認済み — 2026-01 採用決定 / 2026-04 全クライアント移行完了

## コンテキスト

オンプレミスEC2 138台のメール配信基盤を EKS に移行するにあたり、
ワーカーノードの容量タイプ（Spot / On-Demand の比率）を決める必要があった。

経営からの要求と現場制約：

- **コスト削減目標**：3アカウント合計で 3割カット（数百万円規模）
- **削減分は新規事業投資に充当**できるベンチャー企業の構造
- オンプレ時代は貧弱なリソースで大量のインスタンスを並べる構成だったため、
  tmpfs などの I/O 高速化手法が使えなかった
- 配信ワークロードは時間帯で大きく変動（昼70台 / 深夜8台）
- メール配信は Postfix のリトライ機構があり、
  ノード消滅時のキュー消失は S3 退避 + 再送で救済する方針
  （ADR 0005 で詳細設計）

つまり「**コスト削減 + パフォーマンス向上を両立させる構成**」が必要だった。

## 選択肢

### 選択肢 A: Spot 0%（全 On-Demand）
- メリット:
  - 中断リスクなし、最も安定
  - 設計がシンプル
- デメリット:
  - コスト削減目標を達成できない
  - 経営要求に応えられない

### 選択肢 B: Spot 50% / On-Demand 50%
- メリット:
  - ある程度のコスト削減
  - 一定の安定性
- デメリット:
  - 削減効果が中途半端で目標に届かない
  - 設計の複雑さに対するリターンが少ない

### 選択肢 C: Spot 100% + On-Demand フォールバック（採用）
- メリット:
  - 最大限のコスト削減効果
  - 3段階 NodePool でフォールバック可能（small-spot → medium-spot → On-Demand）
  - small/medium インスタンスへの分散で Spot 在庫リスクを軽減
- デメリット:
  - Spot 中断時のリスクを設計で吸収する必要がある
  - tmpfs を採用しているためキュー消失リスクが理論上存在

### 選択肢 D: Spot 100%（フォールバックなし）
- メリット:
  - 最安
- デメリット:
  - Spot 在庫枯渇時に配信停止リスク
  - ベンチャーとはいえ配信品質は譲れない

## 決定

**Spot 100% + 3段階 NodePool による On-Demand フォールバック**を採用。

## 理由

ベンチャー企業として削減分を新規事業投資に直結させる必要があり、
半端な削減ではなく Spot 100% で**安定して経費削減**できる構成が必須だった。

具体的には以下を満たすため：

- **コスト目標達成**：3アカウント合計で 3割カットを実現
- **可用性担保**：3段階 NodePool で Spot 枯渇時も On-Demand にフォールバック
- **パフォーマンス向上の両立**：small/medium Spot で十分なリソースを確保し、
  tmpfs による I/O 高速化が可能になった
- **配信品質維持**：Postfix のリトライ機構に加え、
  ノード消滅時には S3 へ退避して再送する仕組みで対応する
  （詳細設計は ADR 0005）

「コスト削減のためにスペックを落とす」のではなく、
「**コスト削減と同時にパフォーマンスも上がる**」構成にできた点がポイント。

## トレードオフ（諦めたこと）

### Gateway は On-Demand 固定
オンプレとの橋渡しを担う Gateway は中断不可のため、
Spot ではなく On-Demand での運用に限定した。
Gateway 用 On-Demand ノードを別途確保することで、
配信 Pod の Spot 100% 構成と分離している。

### Spot 中断時のキュー消失リスク
tmpfs を採用しているため、Pod 終了時にキュー中のメールが
消失する可能性を受け入れている。
4ヶ月の運用実績では Spot 起因の配信影響は観測されていないが、
根本的な救済設計は ADR 0005 で別途検討する。

### 3段階 NodePool の運用複雑性
NodePool を small-spot / medium-spot / On-Demand の3つに
分割するため、設定ファイルや weight 調整の管理コストが増えた。
詳細は ADR 0004（予定）に分離して記録。

## 結果

### 達成した成果
- **コスト目標達成**：3アカウント合計で 3割カット完遂
- **パフォーマンス向上の両立**：tmpfs 化により I/O 高速化を実現
- **可用性維持**：4ヶ月の運用で Spot 起因の配信影響なし
- **Spot 中断実績**：過去5日間の調査で AWS 側の中断イベント 0回

### 副次的なメリット
- ノードが頻繁に入れ替わるため、OS パッチや AMI の老朽化が自然解消される
- 「ノードはいつでも消える」前提の設計を強制されることで、
  キュー救済システムや expireAfter ズラし運用など、
  耐障害設計が自然に洗練された

## 今後の検証課題

実運用4ヶ月で Spot 中断起因の配信影響は観測されていないが、
tmpfs を使用している以上、理論上のリスクとして以下が残っている：

### Spot 中断時のキュー消失リスク
- **現状**：キュー救済システムは「1,000件超の滞留」を発動条件としており、
  Spot 中断時の即時退避は未実装
- **時間的制約**：
  - AWS Spot 中断通知：2分前
  - K8s の terminationGracePeriodSeconds：現状デフォルト30秒
  - 1,000件の S3 退避実測：1〜2分（20並列時）
- **未確認事項**：
  - Karpenter SQS interruption queue が未設定のため、
    AWS の中断通知を受信できていない可能性
- **対策案**（ADR 0005 で詳細検討）：
  - SQS interruption queue の導入
  - preStop hook での緊急 S3 退避
  - terminationGracePeriodSeconds の調整
